# Exercise: Write Your First Real Prompt

**Module:** 14 — First Useful Workflows

## Learner Objective

Apply the four-part prompt structure to a real task from your own work and identify the human review step.

## What You'll Do

Pick a real task you have on your plate — or coming up this week — and write a prompt for it using the four-part structure from this module.

## The Task

Think of one task in your current work where AI assistance would actually help. It can be small. A summary, a comparison, a first draft — anything from the three workflows in the module.

Now write a prompt for it using this structure:

**Goal:** What do you want the output to be?

**Source:** What material should the AI use? (a document you'd paste or reference, your own notes, nothing external)

**Format:** How should the answer be structured? (length, bullet points vs paragraphs, level of detail)

**Review boundary:** What should the AI not decide for you? What will you review or change yourself before the output is used?

Write it out in full — not just the labels, but the actual prompt you'd send.

Then answer:

**What is the one thing you'd check in the output before you used it or shared it?**

## Completion Criteria

- [ ] The prompt includes all four parts (goal, source, format, review boundary)
- [ ] The task is real — not a generic or made-up example
- [ ] The review boundary explicitly states something the AI should not decide
- [ ] The verification step is specific to the output (not just "I'd read it")

## Common Confusion

**"I can't think of a real task."** Start with the last report you wrote, the last email you drafted, or the last document you had to read quickly before a meeting. Any of those works.

**"My prompt is only two sentences."** That's fine — short prompts can hit all four parts. Length is not the measure. Check that all four elements are present, even implicitly.

## Facilitator Note

This exercise reveals whether the learner has made the connection from concept to their actual work. If the prompt they write is generic ("summarize a document for me"), ask: "What document? For who? In how many words?" Walking them through specificity in real time is more valuable than any module content.
